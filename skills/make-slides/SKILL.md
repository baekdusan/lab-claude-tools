# 발표 슬라이드 생성 — 기존 .pptx 템플릿 채우기 (python-pptx)

연구실/회사의 **공식 .pptx 템플릿 양식을 그대로 유지**하면서 내용만 채워 발표자료를 만드는 스킬. [python-pptx](https://python-pptx.readthedocs.io/)로 템플릿 파일을 열어, 그 안에 정의된 레이아웃과 플레이스홀더(제목 칸·본문 칸 등)에 텍스트를 넣는다. 마스터 슬라이드의 폰트·색·로고·배치가 그대로 보존된다.

핵심 목표: **예쁜 슬라이드가 아니라, ① 템플릿 양식을 깨지 않고 ② 발표자가 말로 풀 수 있는 논리 흐름을 가진 슬라이드.**

## 전제 조건 확인

1. **템플릿 파일**: 이 스킬에는 연구실 공용 기본 템플릿이 번들되어 있다 — `assets/template.pptx` (이 SKILL.md와 같은 스킬 디렉토리 기준). 사용자가 템플릿을 따로 지정하지 않으면 **이 번들 템플릿을 기본으로 사용**한다. 사용자가 다른 `.pptx`(또는 `.potx`) 경로를 주면 그걸로 대체한다. 즉 별도 지정이 없어도 항상 공용 양식으로 발표자료가 나온다.
2. **python-pptx 설치 여부**:
   ```bash
   python3 -c "import pptx; print(pptx.__version__)" || pip3 install python-pptx
   ```

## 작업 흐름 (반드시 이 순서)

### 1단계 — 템플릿 구조 스캔 (가장 중요)

python-pptx로 채우려면 **각 레이아웃의 인덱스와 그 안의 플레이스홀더 idx/type/이름**을 먼저 알아야 한다. 콘텐츠를 짜기 전에 `scripts/inspect_template.py`를 실행해 템플릿 지도를 만든다. 인자를 생략하면 번들된 기본 템플릿(`assets/template.pptx`)을 스캔한다:

```bash
# 기본 번들 템플릿 스캔
python3 scripts/inspect_template.py
# 또는 사용자가 지정한 템플릿
python3 scripts/inspect_template.py "사용자_템플릿.pptx"
```

출력 예시를 보고 "타이틀 레이아웃은 0번, 제목+본문 레이아웃은 1번, 섹션 구분은 2번…" 식으로 매핑 계획을 세운다. **이 지도를 사용자에게도 간단히 보여주면** 어떤 레이아웃을 쓸지 합의하기 쉽다.

### 2단계 — 콘텐츠 아웃라인 합의

입력(텍스트/마크다운/논문 요약/논문)을 슬라이드 구조로 설계해 **텍스트 아웃라인을 먼저 제시**하고 동의를 받는다. 바로 .pptx를 만들지 말 것. 입력이 모호하면 다음을 확인:
1. **청중** — 전공자 세미나 / 비전공 발표 (용어 깊이)
2. **발표 시간** — 슬라이드 수 (대략 1슬라이드 = 1~1.5분)
3. **목적** — 정보 전달 / 설득 / 진행 공유

### 3단계 — 빌드 스크립트 작성·실행

아웃라인 합의 후, 1단계에서 파악한 레이아웃/플레이스홀더 인덱스에 맞춰 빌드 스크립트(`build_deck.py`)를 작성하고 실행한다. `scripts/build_example.py`를 출발점으로 복사해 수정한다.

**기본 정책 — 레이아웃만 빌리고 본문은 새로**: 템플릿에 예시 슬라이드가 들어있어도 그 내용은 쓰지 않는다. 빌드 시작 시 `clear_slides(prs)`로 기존 슬라이드를 모두 비우고(마스터·레이아웃·테마는 보존됨), 레이아웃을 빌려 새 슬라이드를 처음부터 채운다. 사용자가 명시적으로 "기존 슬라이드 뒤에 이어붙여라"라고 하면 그때만 `clear_slides`를 생략한다.

### 4단계 — 결과 확인 안내

출력 `.pptx` 경로를 알리고, 사용자가 PowerPoint/키노트에서 열어 검토하도록 안내. 그림 자리·표 등 손볼 곳을 짚어준다.

## 슬라이드 설계 원칙

1. **한 슬라이드 = 한 메시지**: 제목이 그 슬라이드의 주장이 되도록. "실험 결과"(X) → "제안 기법이 baseline 대비 +2.3점"(O)
2. **불릿은 3~5개, 한 줄 1메시지**: 문단 통째로 넣지 말 것. 발표자가 말로 채울 여백을 남긴다
3. **숫자·고유명사는 슬라이드에, 부연 설명은 발표자 노트(notes_slide)에**
4. **흐름은 Why→What→How→Result**
5. **시각 요소 우선**: 표/도식으로 표현 가능하면 불릿보다 표. 이미지가 필요하면 자리표시자로 표시하고 사용자에게 요청

## 표준 슬라이드 구성

템플릿이 제공하는 레이아웃에 맞춰 조정:
1. **타이틀** — 제목 / 발표자 / 날짜 / 부제 (보통 layout 0)
2. **목차 / 한 줄 요약**
3. **배경·동기 (Why)**
4. **핵심 내용 (What/How)** — 2~5장, 한 슬라이드 한 메시지
5. **결과·근거 (Result)** — 핵심 수치/그림 2~3개
6. **한계·향후 (Discussion)**
7. **결론 / Take-home message**
8. (선택) **Appendix / 참고문헌**

## 논문 발표 매핑 (CPR → 슬라이드)

paper-summary의 CPR 요약을 받으면:

| CPR 요소 | 슬라이드 |
|---|---|
| 한 줄 요약 | 타이틀 부제 + 한 줄 요약 슬라이드 |
| Context | 배경 슬라이드 1장 |
| Problem (Condition + Cost) | 동기/문제 슬라이드 1~2장 — Cost를 "So what?"으로 강조 |
| Response (Claim → How → Evidence) | 핵심 내용 슬라이드 2~4장 |
| 한계 및 열린 질문 | Discussion 슬라이드 |
| 우리 연구와의 관련성 | (내부 세미나면) 마지막 토론 슬라이드 |

## python-pptx 채우기 핵심 패턴

```python
from pptx import Presentation

prs = Presentation("템플릿.pptx")   # 양식 그대로 로드

# 템플릿에 이미 들어있는 더미 슬라이드가 있으면 그걸 쓰거나, 레이아웃으로 새로 추가
title_layout = prs.slide_layouts[0]
body_layout  = prs.slide_layouts[1]

# 타이틀 슬라이드
s = prs.slides.add_slide(title_layout)
s.placeholders[0].text = "발표 제목"      # idx는 inspect 결과 보고 결정
s.placeholders[1].text = "부제 · 발표자 · 날짜"

# 제목 + 본문(불릿) 슬라이드
s = prs.slides.add_slide(body_layout)
s.placeholders[0].text = "제안 기법은 baseline 대비 +2.3점"
body = s.placeholders[1].text_frame
body.text = "첫 번째 포인트"
for line in ["두 번째 포인트", "세 번째 포인트"]:
    p = body.add_paragraph()
    p.text = line
    p.level = 0

# 발표자 노트 (슬라이드엔 키워드, 풀어쓸 말은 여기)
s.notes_slide.notes_text_frame.text = "여기서 청중에게 익숙한 사례로 시작"

prs.save("발표자료.pptx")
```

주의:
- **플레이스홀더 idx는 템플릿마다 다르다** → 반드시 1단계 inspect 결과를 보고 지정. 추측 금지
- 템플릿에 표 스타일이 있으면 `add_table`보다 레이아웃에 포함된 표 플레이스홀더 활용
- **용어 보존**: 모델명·방법명·데이터셋명·메트릭은 영문 원문 유지 (paper-summary 규칙과 동일)
- **수치 정확성**: 반올림·단위 누락 금지
- **그림을 지어내지 말 것**: 없는 그래프/스크린샷은 텍스트 자리표시자로 두고 사용자에게 요청

## 자주 하는 실수 방지

1. **inspect 없이 idx를 추측하지 말 것** — 템플릿마다 플레이스홀더 구성이 다르다. 깨진 슬라이드의 1순위 원인
2. **레이아웃을 직접 그리지 말 것** — 템플릿이 주는 레이아웃만 쓴다. 양식 보존이 목적
3. **슬라이드에 문단 통째로 넣지 말 것** — 그건 문서지 발표자료가 아니다
4. **제목을 라벨로 쓰지 말 것** — "결과"가 아니라 결과의 메시지를
5. **아웃라인 합의 없이 바로 빌드하지 말 것**
6. **슬라이드 수가 발표 시간을 넘지 않게**
